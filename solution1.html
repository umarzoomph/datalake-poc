01

Data Analysis — what's in each file

Four files from three systems were analyzed. Each carries distinct fields and serves a different role in the final unified model.

S3 · Raw

small-video-original-*

Ground-truth ML pipeline output. Contains 7 sampled frames at 1000ms intervals, per-frame detections with bounding boxes, confidence scores, and logo quality metrics. No client context attached.

S3 · Consolidated

21239_6ecc221b-*

Client-enriched derivative. Adds brandDetection[] summaries with engagement and impression valuations, classifierMetadata, and client label mappings. Where valuation math lives.

Elasticsearch

79801-1-*

Client-facing indexed record. Adds social metrics (engagement: 673, impressions: 29,524, videoViews: 4,401), author info, NLP sentiment, and 22 searchable tags that must be preserved.

CSV Export

Export_1344964.csv

97-column reporting output covering all current fields. Performance issue: slow to generate, not real-time. Must be fully supported — this defines the required output contract.

Risk

Duplication at esid level

FACT_POST_METRICS holds exactly one row per esid. Impression counts must never be copied into per-detection rows — this would inflate totals across every brand detected in a post.

Risk

Double-counting detections

The consolidated file's brandDetection[] already aggregates what the raw frames contain. Raw frames are the authoritative source for per-detection granularity — never use both as inputs to the same count.

02

Data Relationship Modeling

Three join keys bind the four sources. SQL Server is the authoritative bridge — it holds the canonical mapping between all three identifiers.

taskId
Links raw message → consolidated S3 file
esid
79801|1|1908676407850377371 · Links ES doc → consolidated via SQL
clientId
21239 · Scopes label mappings and Snowflake partitions
Fallback rule: If the consolidated file is absent for a given taskId, the pipeline uses only the raw message file. Fields sourced from the consolidated file (engagementValuation, impressionsValuation, classifierMetadata, zitem_label_mappings) are set to NULL — not an error condition.
03

End-to-End Architecture

Three-layer data lake on S3 feeds Snowflake via Snowpipe. Ingestion uses S3 event notifications for new files and a Lambda poller for Elasticsearch change detection.

Bronze

Raw landing

  • JSON as-is, no transformation
  • S3 Intelligent-Tiering
  • Partitioned: source/date/clientId
  • Lambda writes on S3 event trigger
  • Glue Catalog registered
  • Immutable — append only
Silver

Normalized & enriched

  • Parquet + Delta Lake format
  • Frames and detections exploded
  • Fallback rule applied
  • ES tags exploded to own table
  • SQL Server mapping joined via JDBC
  • Delta MERGE for upserts
Gold

Snowflake analytics

  • Snowpipe monitors silver prefix
  • COPY INTO fact + dimension tables
  • MERGE on (exposure_id, frame_id)
  • Clustered by client / month / brand
  • View replaces CSV export
  • Result cache for dashboards

05

Example Queries

Brand-level aggregation

SQL · Snowflake
SELECT
  b.brand_class,
  SUM(e.logo_impressions)   AS total_impressions,
  SUM(e.engagement_value)   AS total_engagement_value,
  SUM(e.impressions_value)  AS total_impressions_value,
  AVG(e.confidence)         AS avg_confidence,
  COUNT(DISTINCT e.esid)    AS posts_detected
FROM FACT_EXPOSURE e
  JOIN DIM_BRAND b ON e.brand_key = b.brand_key
WHERE e.client_id = 21239
  AND b.brand_class IN ('kia_motors', 'henry_ford_health_system')
GROUP BY b.brand_class
ORDER BY total_engagement_value DESC;

Asset-level breakdown

SQL · Snowflake
SELECT
  b.brand_class,
  a.asset_class,
  SUM(e.detected_seconds)  AS total_detected_seconds,
  SUM(e.logo_impressions)   AS total_impressions,
  AVG(e.lq_sharpness)       AS avg_sharpness
FROM FACT_EXPOSURE e
  JOIN DIM_BRAND b ON e.brand_key = b.brand_key
  JOIN DIM_ASSET a ON e.asset_key = a.asset_key
WHERE e.client_id = 21239
  AND a.asset_class = 'pole_pad'
GROUP BY b.brand_class, a.asset_class;

Tag-based search — ES parity

SQL · Snowflake
-- Find all posts where StockX appeared on a jersey
SELECT DISTINCT
  e.esid,
  m.author_name,
  m.video_views,
  m.engagement
FROM FACT_ES_TAGS t
  JOIN FACT_EXPOSURE e     ON t.esid = e.esid
  JOIN FACT_POST_METRICS m ON t.esid = m.esid
WHERE t.tag_type  = 'Logo On Asset'
  AND t.tag_value = 'stockx | jersey';

-- Posts where both Kia AND Henry Ford appeared (multi-tag AND)
SELECT esid, COUNT(DISTINCT tag_value) AS matched_tags
FROM FACT_ES_TAGS
WHERE tag_value IN ('kia_motors', 'henry_ford_health_system')
GROUP BY esid
HAVING matched_tags = 2;

Reporting view — replaces CSV export

SQL · Snowflake · VW_REPORT_EXPOSURE
SELECT
  e.esid, b.brand_class AS brand, a.asset_class AS asset,
  c.display_name AS partner, m.author_name, m.impressions,
  m.engagement, m.video_views, e.confidence,
  e.detected_seconds, e.logo_impressions,
  e.engagement_value AS brand_exposure_value,
  e.lq_sharpness, e.lq_centrality, e.lq_clearness,
  e.frame_url, e.thumbnail_url
FROM VW_REPORT_EXPOSURE
WHERE client_id = 21239
  AND brand LIKE '%kia%'
ORDER BY brand_exposure_value DESC;
06

Incremental Update Strategy

Change detection is based on Elasticsearch's lastZmphUpdate field. A DynamoDB watermark table tracks the last-seen timestamp per feed. Lambda polls every 60 seconds and enqueues changed esid values to SQS for batch processing.

01
ES change detected
Poll lastZmphUpdate > DynamoDB watermark
02
Lambda extractor
Fetch changed esid docs, write to S3 bronze/updates/
03
Glue ETL re-run
Re-flatten and enrich only affected esid partition
04
Snowflake MERGE
MATCHED → UPDATE, NOT MATCHED → INSERT
05
Tags refresh
DELETE WHERE esid, re-INSERT updated tag set

MERGE statement

SQL · Snowflake · MERGE on composite key
MERGE INTO FACT_EXPOSURE tgt
USING staging_detections src
  ON tgt.exposure_id = src.exposure_id
 AND tgt.frame_id    = src.frame_id
WHEN MATCHED THEN UPDATE SET
  confidence        = src.confidence,
  engagement_value  = src.engagement_value,
  impressions_value = src.impressions_value,
  is_deleted        = src.is_deleted,
  updated_at        = CURRENT_TIMESTAMP()
WHEN NOT MATCHED THEN INSERT (
  exposure_id, esid, task_id, client_id,
  frame_id, frame_time_ms, brand_key, asset_key,
  confidence, detected_seconds, logo_impressions,
  engagement_value, impressions_value, loaded_at
) VALUES (
  src.exposure_id, src.esid, src.task_id, src.client_id,
  src.frame_id, src.frame_time_ms, src.brand_key, src.asset_key,
  src.confidence, src.detected_seconds, src.logo_impressions,
  src.engagement_value, src.impressions_value, CURRENT_TIMESTAMP()
);
Key design decision: The MERGE key is (exposure_id, frame_id) — a reprocessed detection updates in-place without creating a duplicate row. The composite key guarantees referential integrity even when the same brand is detected in multiple frames of the same post.
07

Optimization

AreaRecommendationRationale
Snowflake clustering CLUSTER BY (client_id, DATE_TRUNC('month', loaded_at), brand_key) Matches query patterns — nearly every dashboard filter starts with client scope, then date range, then brand. Expect 60–80% partition pruning.
S3 storage tiering Bronze: S3 Intelligent-Tiering. Silver: S3 Standard. Bronze objects accessed infrequently after 30 days auto-migrate to cheaper tiers. Silver stays Standard — Glue reads it frequently during incremental jobs.
Snowflake warehouse sizing XS for Snowpipe ingest. S/M for dashboard queries. Snowpipe runs in background — XS is sufficient. Dashboard aggregation queries benefit from result caching; S or M warehouse handles concurrent users without over-provisioning.
Silver partitioning clientId={x}/year={y}/month={m}/ Glue incremental jobs read only the affected partition on reprocessing, keeping DPU costs low. Snowpipe tracks which Parquet files it has already loaded.
Metric fan-out prevention Post metrics stay in FACT_POST_METRICS (one row per esid), joined at query time. If impression counts were copied into each per-detection exposure row, a post with 10 brand detections would report 10× the actual impressions in aggregate queries.
Glue ETL batching SQS queue buffers changed esid set. Glue job triggered when queue depth > 50 or after 5 min TTL. Avoids per-record Lambda invocations at scale. Batch processing is more cost-effective at >1K records/min.
Delta Lake transaction log Use Delta's change data feed for Snowpipe incremental loads. Delta's _change_data feed provides exact list of new/modified Parquet files since last load — eliminates full-prefix scanning by Snowpipe.
Snowflake result caching Enable result caching on reporting virtual warehouse. Dashboard queries hitting the same client + date + brand filters repeat frequently. First query executes; subsequent identical queries return cached results in milliseconds at zero compute cost.
